## Цель

Привести постоянные правила проекта к единой непротиворечивой архитектуре:

- убрать дублирование правил между документами;

- полностью запретить Агенту любые Git-команды;
- закрепить единый жизненный цикл TASK;
- закрепить новый порядок проверки после отчёта;
- оставить один краткий формат ОТЧЁТА АГЕНТА;
- убрать универсальные Codex-специфичные формулировки;
- оставить в БЛОКЕ 3 только последний актуальный статус;
- сохранить логичную навигацию Obsidian;
- не изменить смысл правил карточек товаров, Excel и анализа рынка.

## Обязательное резервное копирование

До первого изменения создать резервные копии всех изменяемых документов.

Папка:

`M:\GitHub\OziWal_AGENT_BACKUP\TASK_008\`

Резервная копия должна находиться вне:

- репозитория;
- Obsidian-хранилища;
- папки `Support`;
- `%TEMP%`.

Если папка уже существует, использовать:

`TASK_008_02`

`TASK_008_03`

Для каждого резервируемого файла проверить:

- существование;
- ненулевой размер;
- совпадение SHA-256 исходного файла и резервной копии.

Если резервное копирование или проверка SHA-256 не выполнены, документы не изменять и завершить задачу как невыполненную.

## Жёсткое правило Git

Зафиксировать в `Support/Правила_агентского_цикла.md` без исключений:

- все операции Git выполняет только Пользователь;
- Агенту запрещены любые Git-команды;
- запрещены `git status`;
- запрещены `git diff`;
- запрещены `git log`;
- запрещены `git add`;
- запрещены Commit;
- запрещены Push;
- запрещены Pull и Fetch;
- запрещено переключение веток;
- запрещено создание веток;
- запрещено изменение Git-конфигурации;
- запрещено обращение к GitHub и другим удалённым репозиториям;
- прямое указание Пользователя внутри обычной TASK не отменяет этот запрет;
- изменение этого правила возможно только через отдельное архитектурное решение Пользователя и новую TASK по изменению постоянных правил.

Удалить из всех документов формулировки:

- `Пользователь обычно сам делает Commit и Push`;
- `Агент не делает Commit и Push без прямого указания`;
- любые другие формулировки, допускающие Git для Агента.

Использовать единое написание:

- `Commit`;
- `Push`;
- `Git`;
- `GitHub`.

## Единый жизненный цикл TASK

В `Support/Правила_агентского_цикла.md` закрепить следующий порядок:

1. Ассистент проверяет актуальную вершину `main` и показывает СТАРТОВЫЙ КОММИТ.
2. Ассистент формирует полную TASK и короткое сообщение запуска.
3. Пользователь публикует TASK в БЛОКЕ 1.
4. Пользователь отправляет короткое сообщение Агенту.
5. Агент выполняет TASK локально без Git и сети.
6. Агент публикует краткий ОТЧЁТ АГЕНТА.
7. При полном выполнении Агент:
   - записывает отчёт в БЛОК 2;
   - записывает предварительный статус в БЛОК 3;
   - очищает содержимое БЛОКА 1 с сохранением оболочки.
8. При частичном выполнении или ошибке БЛОК 1 не очищается полностью.
9. Пользователь выполняет Commit и Push.
10. После получения отчёта Ассистент сразу запрашивает актуальную вершину `main` через GitHub.
11. Ассистент показывает СТАТУС КОММИТА.
12. Ассистент без дополнительного запроса:
    - открывает новый commit;
    - изучает diff;
    - читает изменённые документы;
    - проверяет созданные, удалённые, переименованные и перемещённые файлы;
    - сверяет фактический результат с TASK;
    - выполняет REVIEW.
13. Ассистент принимает одно решение:
    - принять результат;
    - создать новую TASK на исправление;
    - выполнить частичный откат;
    - выполнить полный откат.
14. Завершённая TASK не переписывается задним числом.
15. Все замечания оформляются новой TASK.

## Проверка нового Commit после отчёта

Добавить отдельное точное правило.

После получения ОТЧЁТА АГЕНТА Ассистент:

1. кратко читает отчёт;
2. отмечает только очевидные критические нарушения;
3. сразу проверяет актуальную ветку `main`;
4. не проводит длинную предварительную проверку только по тексту отчёта.

Если новый Commit обнаружен, Ассистент выводит:

СТАТУС КОММИТА

Repository: owner/repository  
Branch/ref: main  
Commit SHA: полный SHA  
Commit message: точное сообщение  
Проверка: выполнена через GitHub

После этого Ассистент сразу выполняет фактический REVIEW.

Если новый Commit не обнаружен, Ассистент выводит:

НОВЫЙ КОММИТ НЕ ОБНАРУЖЕН

Repository: owner/repository  
Branch/ref: main  
Последний известный Commit SHA: полный SHA  
Последнее сообщение Commit: точное сообщение

После этого Ассистент ожидает Commit и Push Пользователя.

Проверка старого Commit вместо нового результата запрещена.

ОТЧЁТ АГЕНТА не является источником истины после Push.

Источником истины после Push являются:

- новый commit;
- diff;
- фактическое содержимое репозитория GitHub.

## Решения после REVIEW

В основном документе зафиксировать четыре допустимых решения:

### Принять

Результат соответствует TASK, критических ошибок нет.

### Новая TASK на исправление

Основной результат полезен, откат не нужен, но необходимы отдельные исправления.

### Частичный откат

Нужно отменить только отдельные ошибочные изменения.

### Полный откат

Результат в целом ошибочен или опасен и требуется возврат к предыдущему состоянию.

Откат выполняет только Пользователь.

Ассистент определяет рекомендуемый объём отката и объясняет причину.

Агент не выполняет Git-откат.

Зафиксировать, что Агент:

- публикует отчёт;
- не выполняет Commit;
- не выполняет Push;
- не принимает окончательное решение о приёмке TASK.

В ограниченном режиме Агент является исполнительными руками Ассистента.

## 1.5. Добавить раздел `Режимы выполнения агента`

Добавить два режима.

### Ограниченный

В ограниченном режиме:

- Ассистент выполняет весь анализ;
- Ассистент принимает все архитектурные и технические решения;
- Ассистент указывает точные файлы, разделы и изменения;
- Агент выполняет только прямо указанные действия;
- Агент не выбирает альтернативную реализацию;
- Агент не выполняет дополнительные улучшения;
- Агент не исправляет побочные проблемы;
- Агент не создаёт дополнительные файлы;
- Агент не изменяет дополнительные файлы;
- Агент не переписывает документы полностью, если достаточно локальной правки;
- при неоднозначности Агент останавливается и сообщает о проблеме.

### Самостоятельный

В самостоятельном режиме:

- Ассистент определяет цель, архитектуру, ограничения и критерии готовности;
- Агент может самостоятельно исследовать связанные файлы;
- Агент может выбирать локальный технический способ реализации;
- Агент может выполнять необходимые проверки;
- Агент может исправлять только ошибки, непосредственно вызванные выполняемой TASK;
- Агент не изменяет общую архитектуру без разрешения;
- Агент не выходит за границы TASK;
- Агент не выполняет Commit;
- Агент не выполняет Push.

Добавить правило:

## 1.10. Формат публикации TASK

Исправить противоречие с `Support/START.md`.

Зафиксировать:

Публикация TASK состоит из двух самостоятельных копируемых блоков:

1. полный текст TASK;
2. короткое сообщение запуска для Агента.

Каждый из этих двух блоков должен быть цельным и готовым для копирования одной кнопкой.

Внутри полного текста TASK запрещено использовать вложенные fenced code blocks, если пользователь явно не попросил иное.

Не писать, что вся публикация должна состоять только из одного блока.

---

## 1.11. Жизненный цикл TASK

Заменить жизненный цикл на следующую последовательность:

1. Ассистент формирует TASK.
2. TASK публикуется в БЛОКЕ 1.
3. Агент выполняет TASK в указанном режиме.
4. Агент публикует отчёт.
5. При успешном выполнении Агент обновляет БЛОК 2, добавляет предварительный статус в БЛОК 3 и очищает содержимое БЛОКА 1 с сохранением структуры.
6. Пользователь выполняет Commit и Push.
7. Ассистент получает последний коммит через GitHub и публикует `СТАТУС КОММИТА`.
8. Ассистент читает изменённые файлы и выполняет REVIEW.
9. Ассистент принимает результат либо создаёт новую TASK.
10. Только после REVIEW задача получает окончательный статус.

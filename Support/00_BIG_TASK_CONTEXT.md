## БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА

# TASK 010 — Система профилей компьютеров и переносимых путей проекта

## Задача

Нужно организовать в проекте `OziWal` универсальную систему локальных путей для нескольких компьютеров.

Пользователь работает с одним репозиторием на трёх компьютерах:

- `PC 1` — домашний компьютер;
- `PC 2` — рабочий компьютер;
- `PC 3` — мобильный компьютер.

Сейчас пользователь работает на `PC 3`.

Текущий локальный путь проекта:

`C:\GitHub\OziWal`

Нужно сделать так, чтобы в будущем пользователь мог дать короткую команду:

`Выставь приоритет ПК на 1`

или:

`Выставь приоритет ПК на 2`

или:

`Выставь приоритет ПК на 3`

После этого агент должен понимать, какой профиль компьютера активен и какой локальный корень проекта использовать.

Система должна быть простой, понятной человеку, безопасной для Git и пригодной как шаблон для других проектов.

## Что прочитать перед началом

Прочитай только:

- `Support/START.md`

Другие документы проекта заранее не изучай.

Разрешается искать по репозиторию абсолютные локальные пути, чтобы определить, где они сейчас используются. Не нужно подробно читать документы, которые не относятся к путям и запуску проекта.

## Текущий контекст

Repository:

`maxrus82gm-svg/OziWal`

Branch:

`main`

Текущий активный компьютер:

`PC 3 — мобильный`

Текущий путь:

`C:\GitHub\OziWal`

Пути для `PC 1` и `PC 2` нельзя придумывать.

Если они однозначно обнаружатся в существующих файлах проекта, можно внести их в таблицу с указанием, откуда они были получены.

Если определить их надёжно нельзя, оставить значения:

- `PC 1: NOT_SET`
- `PC 2: NOT_SET`

## Требуемая архитектура

Нужно реализовать двухуровневую систему.

### 1. Краткая информация в START

В `Support/START.md` добавить короткий раздел:

`Профили компьютеров и локальный путь проекта`

Раздел должен содержать:

- объяснение, что проект может находиться по разным абсолютным путям;
- правило преимущественного использования относительных путей;
- текущий активный профиль;
- ссылку на небольшой отдельный документ с настройками;
- короткий пример команды пользователя:
  `Выставь приоритет ПК на 1`;
- правило для агента, что изменение приоритета означает переключение активного профиля, а не массовую замену путей по всему проекту.

Не нужно превращать `START.md` в большой справочник.

### 2. Небольшой отдельный документ

Создать:

`Support/PROJECT_PC_PROFILES.md`

Документ должен быть коротким и практичным.

Он должен содержать:

- назначение системы;
- таблицу профилей;
- поле активного профиля;
- алгоритм определения корня проекта;
- инструкцию переключения;
- правила для будущих проектов;
- предупреждение не выполнять массовую замену путей без отдельной необходимости.

Пример логики таблицы:

| Priority | Profile ID | Название | Project root | Статус |
|---|---|---|---|---|
| 1 | PC_HOME | Домашний | NOT_SET или найденный путь | configured / not configured |
| 2 | PC_WORK | Рабочий | NOT_SET или найденный путь | configured / not configured |
| 3 | PC_MOBILE | Мобильный | C:\GitHub\OziWal | active |

Активный профиль на момент выполнения:

`ACTIVE_PC_PRIORITY = 3`

Итоговый корень:

`PROJECT_ROOT = C:\GitHub\OziWal`

Названия переменных можно немного улучшить, но они должны оставаться простыми и очевидными.

## Алгоритм разрешения путей

Зафиксировать следующий приоритет:

1. Определить активный профиль по `ACTIVE_PC_PRIORITY`.
2. Получить `PROJECT_ROOT` из таблицы этого профиля.
3. Внутри репозитория использовать относительные пути.
4. Абсолютный `PROJECT_ROOT` использовать только тогда, когда программа или команда действительно требует абсолютного пути.
5. Если путь активного профиля равен `NOT_SET`, агент должен остановиться и запросить путь у пользователя.
6. Агент не должен самостоятельно угадывать отсутствующий путь.
7. Агент не должен заменять все относительные пути абсолютными.
8. Если репозиторий уже открыт из корректной рабочей директории, фактический корень репозитория можно определить через Git и использовать для проверки значения профиля.

## Команда переключения профиля

Документировать, что команда пользователя:

`Выставь приоритет ПК на 1`

означает:

- установить `ACTIVE_PC_PRIORITY = 1`;
- проверить, заполнен ли путь `PC 1`;
- если путь заполнен — использовать его как `PROJECT_ROOT`;
- если путь не заполнен — не угадывать и запросить путь;
- не менять остальные профили;
- не выполнять массовую замену текста во всём репозитории;
- сообщить пользователю, какой профиль и путь теперь активны.

Аналогично для приоритетов `2` и `3`.

## Универсальность для будущих проектов

В `Support/PROJECT_PC_PROFILES.md` добавить небольшой раздел, объясняющий, как переносить систему в другие проекты.

В будущем должны меняться только:

- название проекта;
- список профилей;
- локальные пути;
- активный приоритет.

Сам алгоритм должен оставаться одинаковым.

Не нужно создавать отдельный программный модуль или сложную систему конфигурации.

## Проверка существующих абсолютных путей

Выполни поиск по репозиторию на наличие Windows-путей, например:

- `C:\`
- `D:\`
- `M:\`
- других конструкций вида `<буква>:\`

Цель проверки:

- понять, есть ли уже известные пути для домашнего и рабочего компьютеров;
- выявить места, где абсолютные пути могут мешать переносимости;
- не выполнять автоматическую массовую замену.

Если найдены абсолютные пути:

- перечислить их в отчёте;
- указать файлы, в которых они найдены;
- разделить реальные настройки и обычные примеры в документации;
- менять только те места, которые непосредственно относятся к новой системе профилей или явно создают конфликт с ней.

Не изменять исторические отчёты, примеры или старые записи только ради унификации.

Предпочтительный основной вариант — простая Markdown-конфигурация в `Support/PROJECT_PC_PROFILES.md`.

Не создавать локальный JSON, YAML, PowerShell-скрипт или Python-скрипт без доказанной необходимости.

Если агент считает, что активный профиль лучше хранить в отдельном локальном файле, сначала описать это как рекомендацию в отчёте, но не усложнять текущую реализацию без необходимости.

На данном этапе важнее:

- прозрачность;
- лёгкое ручное переключение;
- понятность для пользователя и Codex;
- возможность перенести схему в другие проекты.

# Правила агентского цикла проекта

> Навигация: [[Support/Карта_документации|Карта документации]] · Связано: [[Support/Правила_запуска_задач|Правила запуска задач]] · [[Support/Рабочая_среда|Рабочая среда]]

## Назначение документа

Этот документ является постоянным источником правил агентского цикла проекта.

Он описывает:

- назначение;
- жизненный цикл TASK;
- роли участников;
- правила выполнения;
- правила приёмки;
- правила внесения изменений.

---

## Единый источник истины

Каждый тип информации хранится только в одном месте.

### Правила единого источника истины

- **Рабочая среда** → `Support/Рабочая_среда.md`
- **Агентский цикл** → `Support/Правила_агентского_цикла.md`
- **Текущая задача** → `БЛОК 1` (в `Support/00_BIG_TASK_CONTEXT.md`)
- **Последний ОТЧЁТ АГЕНТА** → `БЛОК 2` (в `Support/00_BIG_TASK_CONTEXT.md`)
- **Последняя проверка / статус** → `БЛОК 3` (в `Support/00_BIG_TASK_CONTEXT.md`)
- **Постоянные правила проекта** → отдельные документы в `Support/`

**Дублирование постоянных правил между документами запрещено.**

## Роли участников

### Пользователь

- определяет направление проекта;
- утверждает архитектурные решения;
- утверждает изменения процесса;
- контролирует состояние Git;
- выполняет Commit;
- выполняет Push.

### Ассистент

- является архитектором и управляющей системой;
- анализирует проект;
- принимает архитектурные решения;
- формирует TASK;
- определяет границы изменений;
- определяет критерии готовности;
- выполняет проверку через GitHub;
- принимает или отклоняет результат;
- создаёт новую TASK после замечаний.

Ассистент не изменяет локальные файлы проекта напрямую. В ограниченном режиме Ассистент полностью проектирует реализацию и передаёт Агенту точные действия. В самостоятельном режиме Ассистент определяет архитектуру, ограничения и критерии приёмки, а Агент самостоятельно выбирает локальный способ реализации в разрешённых границах.

### Агент-исполнитель

Агент-исполнитель — локальная ИИ-система, имеющая доступ к рабочим файлам проекта и выполняющая TASK. Конкретным агентом может быть Codex, ограниченный локальный агент или другая агентская система.

Codex должен упоминаться только как один из возможных агентов, а не как универсальное название роли.

- работает с локальными файлами проекта;
- выполняет только активную TASK;
- соблюдает установленный режим;
- изменяет только разрешённые файлы;
- проводит только разрешённые проверки;
- публикует отчёт;
- не выполняет Commit;
- не выполняет Push;
- не принимает окончательное решение о приёмке TASK.

В ограниченном режиме Агент является исполнительными руками Ассистента.

## Режимы выполнения агента

### Ограниченный

- Ассистент выполняет весь анализ.
- Ассистент принимает архитектурные и технические решения.
- Ассистент указывает точные файлы, разделы и изменения.
- Агент выполняет только прямо указанные действия.
- Агент не выбирает альтернативную реализацию.
- Агент не выполняет дополнительные улучшения.
- Агент не исправляет побочные проблемы.
- Агент не создаёт и не изменяет дополнительные файлы.
- Агент не переписывает документы полностью, если достаточно локальной правки.
- При неоднозначности Агент останавливается и сообщает о проблеме.

### Самостоятельный

- Ассистент определяет цель, архитектуру, ограничения и критерии готовности.
- Агент может самостоятельно исследовать связанные файлы.
- Агент может выбирать локальный технический способ реализации.
- Агент может выполнять необходимые проверки.
- Агент может исправлять только ошибки, непосредственно связанные с TASK.
- Агент не изменяет общую архитектуру без разрешения.
- Агент не выходит за границы TASK.
- Агент не выполняет Commit и Push.

---

## Правило выбора режима

- Режим, явно указанный в TASK, имеет высший приоритет.
- Если режим в TASK не указан, используется режим текущего рабочего места из `Support/Рабочая_среда.md`.
- Агент не имеет права самостоятельно менять режим выполнения.

---

## Правило контрольной точки отчёта

После публикации TASK начинается этап выполнения.

До получения отчёта агента:

- **БЛОК 1 считается замороженным**;
- требования TASK изменять запрещено;
- критерии выполнения изменять запрещено;
- новые требования в БЛОК 1 добавлять запрещено.

Все возникающие уточнения публикуются только в чате.

**Получение отчёта Агента является контрольной точкой отчёта.**

После получения отчёта Ассистент выполняет проверку.

---

## Неизменяемость БЛОКА 1

После публикации TASK запрещается:

- изменять требования;
- удалять требования;
- добавлять требования;
- изменять критерии приёмки.

Если во время выполнения появляются новые идеи, замечания или дополнительные требования, они оформляются новой TASK после завершения текущего цикла.

---

## Обязательный отчёт агента

После завершения каждой TASK агент обязан опубликовать краткий отчёт в чат.

### Минимальная структура отчёта:

```
━━━━━━━━━━━━━━━━━━━━━━

TASK XXX — выполнено Агентом. Ожидает проверки Ассистентом.

Изменено файлов: N

Изменённые файлы:

• ...

• ...

Кратко выполнено:

• ...

• ...

Обнаруженные проблемы:

• ...

Commit: нет

Push: нет

Ожидает проверки Ассистентом.

━━━━━━━━━━━━━━━━━━━━━━
```

Допускается расширение отчёта дополнительной полезной информацией.

---

## Проверка Ассистента

Работа считается завершённой только после проверки Ассистента.

### Статусы процесса:

- TASK опубликована;
- TASK выполняется;
- контрольная точка отчёта;
- проверка Ассистента;
- Принято.

**или**

- Новая TASK.

**Выполнение задачи не означает её автоматическую приёмку.**

---

## Исправления после проверки

Если после проверки обнаружены замечания:

- опубликованная TASK не изменяется;
- замечания не добавляются в БЛОК 1;
- создаётся новая TASK.

---

## Формат публикации TASK

Публикация TASK состоит из двух самостоятельных копируемых блоков:

1. полный текст TASK;
2. короткое сообщение запуска для Агента.

Каждый блок должен быть цельным и готовым для копирования одной кнопкой.

Внутри полного текста TASK нельзя без явной необходимости использовать вложенные блоки кода с тройными обратными кавычками.

---

## Жизненный цикл TASK

1. Ассистент формирует TASK.
2. TASK публикуется в БЛОКЕ 1.
3. Агент выполняет TASK в указанном режиме.
4. Агент публикует отчёт.
5. При успешном выполнении Агент обновляет БЛОК 2, добавляет предварительный статус в БЛОК 3 и очищает содержимое БЛОКА 1 с сохранением структуры.
6. Пользователь выполняет Commit и Push.
7. Ассистент получает последний коммит через GitHub и публикует СТАТУС КОММИТА.
8. Ассистент читает изменённые файлы и выполняет проверку.
9. Ассистент принимает результат либо создаёт новую TASK.
10. Только после проверки задача получает окончательный статус.

---

## Обновление документа

Документ может обновляться только при принятии архитектурного решения пользователем и утверждении ассистентом.

Изменения фиксируются новой версией документа с описанием изменений.
